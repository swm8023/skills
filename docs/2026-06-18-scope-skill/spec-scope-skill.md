# scope skill — Spec

> Generated by scope skill on 2026-06-18

## Goal

我们要在一个干净的 `skills/` 目录上构建一套自己的 skill 体系，第一阶段需要三个 skill：一个总入口（每次对话强制加载、把 AI 路由到合适的 skill）、一个需求拷问 skill（`scope`，作为几乎所有需求开始前的默认入口）、以及现成可用的 `writing-plans`（直接从原版拷过来）。`scope` 本身的目标是把模糊的用户需求经过一次"对抗式 + 协作式"的访谈榨干歧义、锁定所有决策，最终要么直接生成对话内 plan 立即执行、要么落成 spec 文档进入正式实施流程。

## Decisions

**Q: 整套 skill 体系是新建还是基于原版改造？**
A: 全新构建。`skills/` 是空目录从零开始，`skills-zh/` 和 `skills.bak/` 仅作参考，不参与运行时。

**Q: 第一个新 skill 叫什么？它是 brainstorming 和 grill-me 的结合体，几乎适合于任何需求开始前。**
A: `scope`。一个词同时传达"探索现状 + 锁定边界 + 钉死决策"，工程师/PM 都熟无门槛，不暗示一定产某种文档（对"直接执行"出口友好），与原 brainstorming（散）和 grill-me（攻）形成第三种气质（界）。

**Q: scope 内部走什么流程？**
A: 三阶段：
1. **探索 context** —— 由 scope 自己判断需求规模，小需求跳过、中大需求做主题相关探索；grill 过程中按需补查
2. **grill 式连续提问** —— 一次一题、每题给推荐答案、决策树式深挖、能查 codebase 的不问用户；不再有 brainstorming 原版的"提 2-3 方案"环节，由"推荐答案"机制吸收；不再有 visual companion
3. **询问出口 + 分支处理** —— 决策完成后问用户是否落 spec

**Q: scope 走完后有几种出口？**
A: 两种。
- **不落 spec**：对话里直接生成 plan，用户确认后执行，`docs/` 下零文档
- **落 spec**：生成 `docs/<date>-<slug>/spec-<slug>.md` → self-review → 用户审阅 → 调用 `writing-plans` skill 生成 plan → 执行

**Q: 文档怎么组织？按文档类型还是按主题？**
A: 按主题。`docs/<date>-<slug>/` 下放该主题的所有产物。日期 + 短英文 slug 作为目录名，由 AI 自动生成（如 `2026-06-18-scope-skill`）。同一 topic 的所有相关文档聚在一起，对长生命周期 feature 友好。

**Q: 文档文件名长什么样？**
A: 文件名带 doctype 前缀和 slug，但**不**带日期（日期已经在目录上）。规则：`<doctype>-<slug>.md`，slug 与所在目录的 slug 相同。例如 `docs/2026-06-18-scope-skill/spec-scope-skill.md`、`docs/2026-06-18-scope-skill/plan-scope-skill.md`。这样在多 tab 编辑、grep、最近文件列表等场景下，光看文件名就能知道是"什么主题的什么文档"，不会在多个 topic 同时打开时看到一堆同名 `spec.md` 分不清。

**Q: 落 spec 路径下生成几份文档？**
A: 一份 `spec.md`。最初设计是三件套（brief + spec + plan），但 brief 与 spec 内容重复严重，决定合并 —— brief 的 Q&A 直接写进 spec 的 Decisions 章节，brief 文档移除。

**Q: spec.md 的格式是什么？**
A: 沿用原版风格的 5 节固定结构，外加 Goal / Decisions / Out of Scope：
- Goal（背景 + 目标合并的一两段）
- Decisions（一问一答形式，AI 把零碎的 grill 问题归并成更高阶的 Q+A）
- Architecture
- Components
- Data Flow
- Error Handling
- Testing
- Out of Scope

废弃的备选方案不留、不增加额外章节干扰，简洁优先。

**Q: spec 写完后做什么质量检查？**
A: 保留原版 4 项 self-review：占位符（TBD/TODO/含糊需求）、内部一致性（章节互相矛盾）、scope（是否聚焦到单一计划）、歧义（需求能否被解读成两种含义）。就地修复，无需二次 review。

**Q: 不落 spec 路径下，plan 落到文件吗？**
A: 不落。直接在对话里生成 plan，用户确认后执行，`docs/` 下零文档。

**Q: plan 由谁负责？格式是什么？**
A: 由独立的 `writing-plans` skill 负责，scope 不管 plan 格式细节。`writing-plans` 直接从原版（`skills.bak/writing-plans/`）拷贝过来，不改动。

**Q: scope 如何被触发？**
A: 通过总入口 skill 强制路由。原版 `using-superpowers` 复制并改写为新名字 `auto-using-skills`，作为每次对话强制加载的总入口，规定"1% 适用就必须调用 Skill 工具"，把"加 feature/修 bug/改东西"类话题自动路由到 scope。

**Q: 总入口 skill 叫什么？**
A: `auto-using-skills`。比 `entrypoint` 更具描述性，名字本身就传达了它的机制 —— 自动调用 skill。

**Q: auto-using-skills 相对原版 using-superpowers 改了什么？**
A:
- **保留**：1% 强制调用规则、指令优先级三层（用户 > skill > 默认 system）、红色警示表（"自我合理化"念头对照表）、skill 类型刚性/灵活区分、"指令说做什么不是怎么做"、announce skill 使用
- **改**：所有 "Superpowers"/"brainstorming" 字眼改成新体系语言；流程图 "Already brainstormed?" 节点改为 "Already scoped?"；skill 优先级章节改成 "scope 优先"
- **砍**：跨平台说明段落（Copilot/Gemini）、platform adaptation 段、references/ 目录引用

## Architecture

```
.claude (repo root)
├── skills/                       ← 运行时使用的 skill 集
│   ├── auto-using-skills/        ← 总入口（每次对话强制加载）
│   │   └── SKILL.md
│   ├── scope/                    ← 需求拷问 + spec 落地
│   │   └── SKILL.md
│   └── writing-plans/            ← 从原版拷贝，未改动
│       ├── SKILL.md
│       └── plan-document-reviewer-prompt.md
├── skills-zh/                    ← 中文翻译版（参考资料，不运行）
├── skills.bak/                   ← 英文原版备份（参考资料，不运行）
└── docs/
    └── <YYYY-MM-DD-slug>/
        ├── spec-<slug>.md         ← 主题型组织，文件名带 slug 不带日期
        └── plan-<slug>.md
```

## Components

**auto-using-skills**

- 角色：所有对话的强制入口
- 输入：用户消息
- 输出：决定是否调用某个具体 skill；如果调用，announce 使用
- 触发逻辑：每条用户消息进入时检查"是否有 skill 1% 可能适用"
- 关键约束：用户的明确指令永远优先于本 skill；与默认 system prompt 冲突时本 skill 覆盖默认

**scope**

- 角色：需求拷问器，几乎所有需求开始前的默认入口
- 输入：用户的初始需求（无论模糊还是清晰）
- 输出：对话里的"已确认决策集合"，再加上 spec.md（仅落 spec 路径下）
- 内部三阶段：判断规模 → grill → 分支处理
- 终态：要么对话内 plan 立即执行，要么落 spec 后调用 writing-plans

**writing-plans**

- 角色：把 spec 转成可执行的 plan
- 输入：spec.md
- 输出：plan.md（按原版格式，task/step/code/expected output）
- 状态：从原版直接拷贝，第一阶段不修改

## Error Handling

**scope 过程中的失败模式**：

- **用户跳过 grill 直接要求执行** —— 尊重用户指令（指令优先级最高），但提示"未走完决策树，可能有未确认假设"
- **codebase 探索失败 / 文件不存在** —— 不阻断流程，把"我没找到现状信息"作为一项明确假设留在 spec 的 Goal 里
- **grill 决策树过深，超过 ~10 轮提问** —— scope 应主动评估"是否需求过大、应该拆子项目"；如是，提示用户分解
- **用户审阅 spec 后要求大改** —— 回到 grill 阶段，更新决策，重写 spec，重跑 self-review

**spec self-review 的失败模式**：

- 4 项检查（占位符 / 内部一致性 / scope / 歧义）就地修复，不做二次 review
- 如果 self-review 发现 scope 过大（应拆子项目），暂停并询问用户

**auto-using-skills 的失败模式**：

- 用户的明确指令与 skill 规则冲突 —— 一律遵循用户指令（CLAUDE.md/直接请求等显式指令是最高优先级）
- skill 不存在 / 文件读取失败 —— fallback 到默认行为，不阻断对话

## Testing

**测试策略**：本阶段不写自动化测试，通过实际使用验证。

**验收 case 1：scope skill 自我验证**

- 输入：今天这场关于"创建 scope skill"的对话内容
- 期望产物：`docs/2026-06-18-scope-skill/spec-scope-skill.md`（即本文档）
- 验证：本文档存在、5 节结构齐全、Decisions 反映了所有 grill 出来的决策、self-review 4 项通过

**验收 case 2：完整链路 dry run**

- 输入：用户提一个新需求（任意小 feature）
- 期望路径 1（不落 spec）：scope grill → 询问 → 用户选不落 → 对话里生成 plan → 执行 → `docs/` 下没有该 topic 目录
- 期望路径 2（落 spec）：scope grill → 询问 → 用户选落 → 生成 spec.md → self-review → 用户审阅 → writing-plans 触发 → 生成 plan.md → 执行

**验收 case 3：auto-using-skills 强制路由**

- 输入：用户对话中提到"加个 feature"或"修复 bug"
- 期望：auto-using-skills 自动检测并 announce 调用 scope，不需要用户手动 `/scope`

**反例 / 失败信号**：

- 用户提需求后 AI 没调 scope 直接回答 → auto-using-skills 失效
- scope 一次问多个问题 → 流程退化
- spec 中出现 TBD/TODO → self-review 没起作用
- 文档落在 `docs/specs/` 而不是 `docs/<topic>/` → 主题型组织失效

## Out of Scope

- **第二阶段 skill** —— TDD、debugging、code-review、handoff 等先不做，等第一阶段验证可行后再增量加
- **plan 格式优化** —— writing-plans 直接拷原版，不在本次重写
- **跨平台兼容** —— 砍掉 Copilot/Gemini/Codex 的工具映射，本阶段只针对 Claude Code
- **visual companion** —— 不引入浏览器视觉辅助
- **brainstorming 原版的"提 2-3 方案"机制** —— 由 grill 的"推荐答案"吸收
- **brief.md 文档** —— 已合并进 spec.md 的 Decisions 章节
- **issue tracker / PRD 流程** —— 本阶段是单人本地开发体系，不接 GitHub issue
- **自动化测试 skill** —— 第一阶段靠人工验收
