# debug

## 概览

debug 是处理 bug 的默认入口。目标不是尽快改代码，而是用最短路径获得一个可信答案：问题是什么、为什么发生、怎样证明它被修好。

如果用户要的是新功能、产品行为设计、需求澄清或非 bug 改动，使用 `scope`。如果用户描述的是"坏了、报错、测试失败、变慢、偶发失败、行为不符合预期"，使用本 skill，不要先走 `scope`。

## 铁律

```
NO FIXES WITHOUT A VALIDATION SIGNAL AND ROOT CAUSE INVESTIGATION
```

没有复现、没有证据、没有可证伪假设时，不要提出修复方案。违反流程的字面规定，就是违反调试精神。

禁止：

- 看到症状就改代码。
- 同时改多处再跑测试。
- 在没有复现的情况下推测原因。
- 用 workaround 代替 root cause。
- 在两次失败修复后继续"再试一个"。

## 工作流

### 1. 明确问题与验证信号

先把问题定义清楚，再选择一种能验证问题存在/消失的信号。不要一上来就写测试或改代码。

先确认：

- **实际行为**：用户看到了什么错误、异常输出、慢时序或错误状态？
- **预期行为**：系统本来应该怎样？
- **触发条件**：输入、账号、环境、配置、版本、时间窗口、操作步骤是什么？
- **影响范围**：只影响一个路径，还是多个调用方 / 平台 / 数据集？
- **最近变化**：代码、依赖、配置、数据或部署是否刚变过？

然后选择最便宜、最可靠的验证信号：

1. 失败测试：unit、integration、e2e 都可以。
2. 命令或脚本：curl、CLI fixture、headless browser。
3. 真实 artifact：HAR、payload、event log、日志片段、截图、录屏。
4. 临时 harness：只启动最小系统子集。
5. 对照信号：旧版/新版、配置 A/B、数据集 A/B。
6. 人工复现：必须人手操作时，把步骤和需要捕获的输出结构化。

验证信号不一定一开始就是自动化测试，但必须能支撑判断。能自动化就自动化；不能自动化就先把 artifact 固定下来。

非确定性 bug 的目标不是一次干净复现，而是提高复现率。循环、多跑、加压力、缩小时序窗口，直到复现率足以支持判断。

如果拿不到任何验证信号，停下来说明已尝试的方法，并向用户要可复现环境、捕获 artifact，或临时 instrumentation 许可。没有验证信号，不进入假设阶段。

### 2. 复现并锁定现象

用第 1 阶段选定的验证信号看着 bug 出现。确认：

- 失败模式就是用户描述的那个 bug，不是附近的另一个失败。
- 多次运行能复现，或非确定性 bug 的复现率足以支持调试。
- 已捕获精确症状：错误消息、错误输出、慢时序、错误状态。

错的 bug 会导致错的 fix。不要在这一步含糊。

### 3. 追根因

先理解数据流和控制流，再形成修复方案：

1. 完整阅读 error、warning、stack trace、错误码与路径。
2. 检查最近改动：diff、commit、依赖、配置、环境。
3. 追踪错误值从哪里来，被谁传下去，在哪个边界变坏。
4. 对多组件系统，在边界记录输入、输出、配置传播和状态。
5. 如果错误出现在调用栈深处，沿调用链向上追到最初触发点。

不要只在症状出现的位置修复，除非已经证明那里就是源头。

### 4. 做模式分析

在提出修复前，先找可工作的相邻模式：

- 同一 codebase 里有没有类似但正常工作的实现？
- 参考实现是否被完整阅读，而不是凭印象套用？
- 好的路径和坏的路径有哪些差异？
- 依赖、配置、初始化顺序、环境假设有什么不同？

列出差异，不要预设"这个小差别不可能有影响"。

### 5. 形成假设并验证

生成 3-5 个有排序的可证伪假设。每个假设都必须给出预测：

> 如果 X 是原因，那么改变 Y 会让 bug 消失，或改变 Z 会让 bug 更明显。

把排序后的假设列表给用户看；如果用户 AFK，按你的排序继续。每轮只测试当前 top 1 假设。

instrumentation 必须对应某个预测。一次只改一个变量。

工具偏好：

1. 能用 debugger/REPL 就用，断点胜过大量日志。
2. 在能区分假设的边界打有针对性的日志。
3. 不要"全部打日志再 grep"。

给每条临时日志加唯一 tag，例如 `[DEBUG-a4f2]`，收尾时用同一个 tag 清理。

性能 regression 先建立 baseline 测量，再定位。优先 profiler、query plan、计时 harness、bisect。不要先靠日志猜性能原因。

### 6. 提交诊断结论与修复计划

在写任何生产修复代码之前，先把诊断结论和修复计划提交给用户确认。这是修复前的决策 checkpoint；没有用户确认，不进入修复阶段。

给用户的确认内容必须包含：

- **根因**：具体是哪一层、哪个条件、哪条数据流或哪个调用关系导致问题。
- **证据**：哪个验证信号、日志、断点、diff、trace 或实验支持这个结论。
- **修复方案**：准备改哪里、为什么这样改、是否有替代方案被排除。
- **验证计划**：如何证明 bug 修好了；哪些测试、脚本、人工步骤或观察信号会被重复运行。
- **风险与验证**：可能影响哪些路径，修复后要跑哪些检查。

如果 root cause 仍只是猜测，明确说“还不能修”，回到假设验证。不要把不确定的猜测包装成修复计划。

### 7. 用 TDD 修复

在修复前写回归测试，但只写在正确 seam 上。正确 seam 是能以真实 bug 模式触发问题的调用点；太浅的测试只会制造虚假信心。

用户确认修复方案后，调用 `test-driven-development`。本阶段已有的最小 repro / 回归测试就是 RED 输入；仍要按 TDD 验证它先失败，再写最少修复代码让它通过。

如果没有正确 seam，这本身就是发现：架构阻止这个 bug 被锁死。记录下来，修复后再提出架构改进建议。

执行顺序：

1. 把最小 repro 转成失败测试或可重复脚本。
2. 看着它失败。
3. 实施单一 root-cause 修复。
4. 看着它通过。
5. 重新跑原始验证信号，确认用户场景不再复现。
6. 跑相关测试，确认没有回归。

如果修复无效，停下来回到阶段 2 或阶段 4，不要在失败 fix 上继续叠补丁。三次修复失败后，先质疑架构或根本假设，并和用户讨论后再继续。

### 8. 收尾验证

宣布完成前确认：

- 原始 repro 不再复现。
- 回归测试或可重复脚本通过。
- 相关测试通过。
- 所有 `[DEBUG-...]` instrumentation 已清理。
- 一次性 harness 已删除，或挪到明确标记的位置。
- 最终 root cause 和被证实的假设写进 commit/PR message 或最终说明。

## 方法准则

### 调用栈深处的错误

如果错误出现在底层函数、系统命令、数据库、文件路径或网络边界，不要只修出错点。沿调用链往上问：

- 谁调用了这里？
- 传入的值是什么？
- 这个值最早在哪里被创建或破坏？
- 为什么上游允许这个值进入系统？

源头修复优先于症状修复。

### flaky / race / timeout

不要猜 sleep 时间。等待真实条件：

- 等事件出现，而不是等 200ms。
- 等状态变为 ready，而不是等初始化"应该完成"。
- 等文件存在、计数达到、队列为空或请求完成。
- 如果必须等待固定时间，先等待触发条件，再说明该时间来自已知时序，而不是猜测。

### 纵深防御

如果 root cause 是非法数据、危险状态或错误环境，仅在一处加检查通常不够。沿数据流增加防御：

- 入口校验：API 边界拒绝明显非法输入。
- 业务校验：操作前确认状态对该行为成立。
- 环境守卫：测试、生产、临时目录、权限等上下文约束。
- 取证埋点：在危险操作前保留必要上下文，帮助下一次定位。

### 人工复现

如果必须由用户点击或观察，把人工步骤结构化：

1. 写清每一步用户要做什么。
2. 明确要捕获什么输出、错误、截图或时间点。
3. 把用户反馈转成可解析的事实，再回到验证信号与假设验证流程。

## 警示信号：看到就停下

以下念头意味着你正在猜：

| 念头 | 现实 |
| --- | --- |
| "先临时修一下" | 这是 workaround，不是 root cause。 |
| "八成是 X，我直接改" | 没有预测和证据就不是假设。 |
| "一次改多处省时间" | 你将无法知道哪一处有效。 |
| "问题简单，不需要流程" | 简单 bug 也有 root cause。 |
| "测试后面再补" | 没有失败测试，无法证明修复针对原 bug。 |
| "再试一个 fix" | 两次失败后要重新调查；三次失败后质疑架构。 |
| "日志全打上再说" | instrumentation 必须服务于具体预测。 |
| "这里太难测" | 可能是 seam 不对，也可能是设计在阻止验证。 |
| "用户会同意这个修法" | 先列出根因和修复方案，等用户确认。 |

用户说 "Stop guessing"、"Will it show us...?"、"Is that not happening?" 或类似话时，立即回到 root cause 调查。
