# UI Prototype

在单个路由上生成**若干个截然不同的 UI 变体**，通过一个浮动的底部 bar 切换。用户在浏览器里翻看各个变体，挑一个（或者从每个里偷一些片段），然后把其余的丢掉。

如果问题关乎逻辑/状态而不是某样东西看起来如何 —— 走错分支了。请使用 [LOGIC.md](LOGIC.md)。

## 何时这是合适的形态

- "这个页面应该长什么样？"
- "在定下来之前，我想看几个 dashboard 的备选方案。"
- "为 settings 页换一个不同的布局试试。"
- 任何在用户脑子里要花一整天在三个模糊 mockup 之间挑选的场景。

## 两种子形态 —— 强烈优先选择子形态 A

UI prototype 在**贴着 app 的其它部分**展示时要好评判得多 —— 真实的 header、真实的 sidebar、真实的数据、真实的密度。一个孤立的临时路由就是真空：每个变体单独看都还行。只要存在一个合理的现有页面可以承载这些变体，就默认走子形态 A。只有在 prototype 真的找不到附近的容身之处时，才考虑子形态 B。

### 子形态 A —— 对现有页面的调整（首选）

路由已存在。变体在**同一路由上**渲染，由 `?variant=` URL 搜索参数控制。已有的数据获取、参数和鉴权全部保留 —— 只有渲染部分被替换。这是默认选项；除非有特殊理由，否则就选它。

如果 prototype 是为某个还没有页面、但*会自然地嵌入某个页面之内*的东西做的（dashboard 的一个新区域、settings 屏幕上的一张新 card、一个现有 flow 中的新一步） —— 那仍然属于子形态 A。把变体挂在宿主页面里。

### 子形态 B —— 新页面（最后手段）

只有当被原型化的东西真的没有任何现有页面可以容身时才使用 —— 例如一个完全新的顶级界面，或一个不能合理嵌入任何地方的 flow。

按项目已有的路由约定创建一个**临时路由** —— 不要发明新的顶级结构。命名要让它一眼就是 prototype（例如在路径或文件名中包含 `prototype` 这个词）。同样使用 `?variant=` 模式。

在选定子形态 B 之前，做一次理智检查：真的没有任何现有页面可以把它嵌进去吗？空路由会掩盖那些被填充页面会暴露出来的设计问题。

在两种子形态中，浮动的底部 bar 都是相同的。

## 流程

### 1. 写下问题并确定 N

默认 **3 个变体**。超过 5 个就不再是截然不同，而开始变成噪音 —— 上限就到这里。

把计划用一行写下来，放在 prototype 所在位置或文件顶部的注释里：

> "settings 页的三个变体，通过 `?variant=` 切换，挂在已有的 `/settings` 路由上。"

无论用户是不是来推翻你的，这都管用。

### 2. 生成截然不同的变体

起草每个变体。让每一个都满足：

- 页面的目的，以及它能拿到的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、纯 CSS，无所谓哪种）。
- 一个清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体之间必须**结构上不同** —— 不同的布局、不同的信息层级、不同的主要 affordance，而不仅是不同的颜色。三个微调过的 card 网格不是 UI prototype，那是壁纸。如果有两份草稿太像，就用明确的"不要使用 card 网格"的指引把其中一个重做。

### 3. 把它们串起来

在路由上创建一个单一的切换器组件：

```tsx
// pseudo-code — adapt to the project's framework
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对子形态 A（现有页面）：把所有已有的数据获取保留在切换器之上；每个变体只切换被渲染的子树。

对子形态 B（新页面）：`/prototype/<name>` 下的临时路由挂载同一个切换器。

### 4. 构建浮动切换器

在屏幕底部居中的位置放一个固定定位的小 bar，包含三个部分：

- **左箭头** —— 循环到上一个变体（环绕）。
- **变体标签** —— 显示当前的变体 key，如果变体导出了一个名字，也一并显示。例如 `B — Sidebar layout`。
- **右箭头** —— 向前循环（环绕）。

行为：

- 点击箭头会更新 URL 搜索参数（使用 framework 的 router —— Next 上用 `router.replace`，React Router 上用 `navigate`，等等），这样变体可分享、刷新后状态也稳定。
- 键盘：`←` 和 `→` 方向键也可以循环。当某个 `<input>`、`<textarea>` 或 `[contenteditable]` 处于聚焦状态时，不要拦截方向键。
- 在视觉上要和页面区分开（例如高对比度的胶囊形、淡淡的阴影），让它一眼就明显不是被评估的设计的一部分。
- 在生产构建中隐藏 —— 用 `process.env.NODE_ENV !== 'production'` 或等价的检查做门控，这样一次跑偏的 prototype merge 也不会把这个 bar 发到用户那里。

把切换器放进一个共享组件中，这样两种子形态都能复用。把它放在项目里共享 UI 所在的位置。

### 5. 交付

把 URL（以及 `?variant=` 的各个 key）暴露出来。用户会在腾出空时翻看。有意思的反馈通常是 **"我想要 B 的 header 配上 C 的 sidebar"** —— 那才是他们真正想要的设计。

### 6. 记录答案并清理

一旦某个变体胜出，就写下是哪一个、为什么（commit message、ADR、issue，或者如果是 AFK 运行而用户还没回应时就在 prototype 旁边放一个 `NOTES.md`）。然后：

- **子形态 A** —— 删掉落败的变体和切换器；把胜者并进现有页面。
- **子形态 B** —— 把胜出的变体提升为真正的路由，删掉临时路由和切换器。

不要把变体组件或切换器留在那里。它们会迅速腐烂，并让下一个读者困惑。

## 反模式

- **只在颜色或文案上不同的变体。** 那是微调，不是 prototype。真正的变体在结构上是有分歧的。
- **变体之间共享了过多代码。** 共享一个 `<Header>` 没问题；共享一个 `<Layout>` 就违背了初衷。每个变体都应该可以自由地把布局丢掉。
- **把变体连到真实的 mutation 上。** 只读的 prototype 就够了。如果某个变体需要 mutate，把它指向一个 stub —— 问题是"这看起来应该是什么样"，而不是"backend 能不能跑"。
- **直接把 prototype 推到生产。** 变体代码是在 prototype 约束下写出来的（没有 test、错误处理极少）。在并入时要正经地重写一遍。
